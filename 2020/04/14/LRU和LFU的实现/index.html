LRU和LFU的实现

发表于 2020-04-14 | 分类于 数据结构与算法

LRU和LFU

​ 当我们向某个服务器请求资源时，服务器接收到请求有时并不会在服务器直接在外存内查找数据，因为磁盘IO是很慢的，这时候就需要内存作为一层缓冲，储存经常要用到的数据。具体就是服务器会先在内存中查询数据，如果命中，返回给用户数据，如果没有命中再去数据库查。而内存空间比起硬盘空间，因为工程造价问题，往往是比较小的比较宝贵的，对内存的管理清理就是一项重要的工作。 LRU和LFU都是经常用的内存管理策略。

LRU

LRU是Least Recently Used的缩写，即最近最少使用，是一种常用的页面置换算法，选择最近最久未使用的页面予以淘汰。该算法赋予每个页面一个访问字段，用来记录一个页面自上次被访问以来所经历的时间 t，当须淘汰一个页面时，选择现有页面中其 t 值最大的，即最近最少使用的页面予以淘汰。

LFU

LFU（least frequently used (LFU) page-replacement algorithm）。即最不经常使用页置换算法，要求在页置换时置换引用计数最小的页，因为经常使用的页应该有一个较大的引用次数。但是有些页在开始时使用次数很多，但以后就不再使用，这类页将会长时间留在内存中，因此可以将引用计数寄存器定时右移一位，形成指数衰减的平均使用次数。

​ 简单来说，LRU就是当内存达到一定的值，淘汰当时最近最长时间没有被使用的数据，而LFU就是淘汰最近使用频率最低的数据。

下面用java语言分别实现这两种算法。

LRU

运用你所掌握的数据结构，设计和实现一个 LRU (最近最少使用) 缓存机制。它应该支持以下操作： 获取数据 get 和 写入数据 put 。

获取数据 get(key) - 如果密钥 (key) 存在于缓存中，则获取密钥的值（总是正数），否则返回 -1。
写入数据 put(key, value) - 如果密钥不存在，则写入（设置或插入）其数据值。当缓存容量达到上限时，它应该在写入新数据之前删除最久未使用的数据值，从而为新的数据值留出空间。

进阶:

你是否可以在 O(1) 时间复杂度内完成这两种操作？

​ 要求如上，很明显能在在O(1)时间复杂度内命中目标的数据结构那就是哈希表了，但是哈希表本身是无序的，无法表示数据之间的先后问题，然而LRU要求就是数据之间是按照使用时间的顺序的。那么，可以借助一个线性储存结构灵活的表示数据之间的顺序，最好能快速的删除数据，双向链表是不二之选。为啥单向链表不合适，因为单向链表删除节点时，有一个赋值删除的过程，需要一点技巧啊，不如双向链表表达更直接清晰。

​ 这样，基本的框架就确定了，哈希表加双向链表，哈希表的key-value结构，key明显要储存数据的key，而value要储存的是链表节点，而链表节点储存的数据key和value(后面会解释为什么不能直接储存value)。为了加深理解，我决定从头开始造轮子。

先造节点类，由于是双向链表，每个节点除了key value之外，还应该有一个头指针，一个尾指针。

再造双向链表类，双向链表有一个头结点，一个尾节点，还得有个容量，来实时掌握储存的数据量，以防数据超过限制，这个属性放在最后的LRU类中也可。这个类还需要实现一个头插(加入的数据总是最新的，尾插没有意义)和一个删最后节点的方法(方便删除要淘汰的数据)。

最后，LRU类就可以开写了。要注意，不光我们查询这个数据的时候，要把这个数据在链表中的位置提前，当我们对这个数据重新赋值时，也是使用了这个数据，也要把数据提前。值得注意的是，如果链表中的储存的是value的话，当在链表中删除这个节点时，哈希表是无法得到删除的数据是什么的，哈希表中也就无法对这个数据进行删除了。

LFU

然而，LRU这种用数据的未被使用时间长短来淘汰数据有时候是有问题的，比如说一个相当冷门的数据，就有人恶意攻击，访问些冷门数据，我们的内存岂不是要储存一堆冷门数据了？这时候，使用用频率去淘汰数据，显得更合理一些，这就是LFU。

请你为 最不经常使用（LFU）缓存算法设计并实现数据结构。它应该支持以下操作：get 和 put。

get(key) - 如果键存在于缓存中，则获取键的值（总是正数），否则返回 -1。
put(key, value) - 如果键不存在，请设置或插入值。当缓存达到其容量时，则应该在插入新项之前，使最不经常使用的项无效。在此问题中，当存在平局（即两个或更多个键具有相同使用频率）时，应该去除 最近 最少使用的键。
「项的使用次数」就是自插入该项以来对其调用 get 和 put 函数的次数之和。使用次数会在对应项被移除后置为 0 。

进阶：
你是否可以在 O(1) 时间复杂度内执行两项操作？

要求大同小异，哈希表和双向链表的结合就可以达到目的，但是明显一个双向链表已经解决不了问题了，我们必须对数据节点按照频率进行分类，同样的频率放进同样的链表。如果这样的话， 我们还需要一个频率和链表的映射，那么我们还需要一个哈希表来储存频率和链表。

先造节点类，每个节点对象刚被创建，频率默认为1，其余和LRU的节点类一致。

再造双向链表类，这次没有再在双向链表类中加size属性，原因是这次要用很多个链表， 再把size放在这，每次比较岂不是要遍历所有的链表，拿到各自的size再加和？那太费事了，干脆就放在最后的LFU类中吧。

最后的LFU类，最核心的方法是freqInc方法，这个方法实现了对节点的频率提升。

总结

双向链表和哈希表的结合，实现LRU和LFU。
